### 1. 금융권 유스케이스별 AI 인프라 최적화 포인트

각 서비스의 성격에 따라 GPU 자원을 배분하고 튜닝하는 전략이 완전히 달라져야 합니다.

#### Case A: 대고객 챗봇 서비스 (B2C)

- **인프라 특성:** 짧은 프롬프트(질문)가 동시다발적으로 들어오며, 트래픽의 변동성(Spike)이 큽니다. 가장 중요한 지표는 **응답 지연 시간(TTFT) 최소화**와 **동시 접속자 처리량(Throughput)**입니다.
    
- **최적화 포인트:**
    
    1. **Continuous Batching:** 트래픽이 몰릴 때 사용자의 질문을 고정된 크기로 묶일 때까지 기다리지 않고, GPU의 빈 슬롯이 생기는 즉시 동적으로 밀어 넣어 전체 시스템의 처리량을 극대화합니다.
        
    2. **Quantized KV Cache:** 수만 명의 고객 대화 기록(Context)을 GPU 메모리에 유지하려면 VRAM이 턱없이 부족합니다. KV 캐시를 INT8 또는 INT4로 양자화하여 동일한 GPU에 2~3배 더 많은 세션을 올릴 수 있게 해야 합니다.
        
    3. **llm-d gateway:** 고객 프롬프트 내 주민등록번호, 계좌번호 등의 민감정보(PII)를 GPU로 넘어가기 전에 마스킹(Masking)하고, API 속도 제한(Rate Limiting)을 걸어 인프라 과부하를 막는 관문 역할을 반드시 세워야 합니다.
        

#### Case B: RM 기업여신심사 자동화 (B2B/내부망)

- **인프라 특성:** 수백 페이지에 달하는 기업 재무제표, 감사보고서, 약관 등을 한 번에 입력해야 하므로 **초장문 컨텍스트(Long-context) 처리 능력**이 필수입니다. 숫자에 민감하므로 할루시네이션(환각)이 없어야 합니다.
    
- **최적화 포인트:**
    
    1. **FlashAttention (또는 TriAttention):** 긴 문서가 입력될 때 $O(N^2)$로 폭발하는 어텐션 연산 메모리를 제어하고 처리 속도를 높이기 위해 필수적으로 활성화해야 합니다.
        
    2. **Chunked Prefill:** 거대한 문서를 읽어 들이는 작업(Prefill)이 GPU를 너무 오래 독점하면, 다른 RM 직원이 보낸 짧은 질문의 응답이 멈춰버립니다. 긴 입력을 여러 청크로 쪼개어 스케줄링함으로써 시스템 공정성을 유지해야 합니다.
        
    3. **MIG / DRA:** 망분리된 온프레미스 환경에서 부서별로 GPU를 나누어 쓰거나(MIG), 야간에 수만 건의 일괄 심사(Batch)를 돌릴 때 유휴 GPU 자원을 동적으로 끌어다 쓰는(DRA) 자원 스케줄링이 중요합니다.

### 2. TriAttention 논문 분석 (MIT, NVIDIA 등 / 2026.04 발표)

제공해주신 링크(arXiv:2604.04921)의 논문은 **"TriAttention: Efficient Long Reasoning with Trigonometric KV Compression"**으로, 방대한 문서를 처리해야 하는 '여신심사 자동화'와 같은 업무에 혁명적인 돌파구를 제시한 최신 연구입니다.

#### ① 기존 방식의 한계

기존의 KV 캐시 압축 기술(SnapKV, H2O 등)은 LLM이 어느 토큰에 집중했는지 나타내는 '어텐션 점수'를 기반으로 덜 중요한 토큰을 캐시에서 삭제(Eviction)했습니다. 이때 위치 정보가 이미 반영된 상태(Post-RoPE)의 점수를 사용했는데, 이는 긴 문맥에서 복잡한 수학적 추론이나 논리 전개를 할 때 중요한 정보를 실수로 날려버려 정확도를 치명적으로 떨어뜨리는 원인이 되었습니다.

#### ② 접근 방식의 변화: Pre-RoPE 공간과 삼각함수

연구진은 발상을 전환하여 위치 임베딩이 적용되기 전(**Pre-RoPE**)의 공간을 분석했습니다. 여기서 Query(Q)와 Key(K) 벡터들이 특정한 '중심점(Center)' 주변에 고도로 밀집되어 있다는 사실을 발견했습니다. 이 중심점들은 Query가 어떤 거리(Distance)에 있는 Key를 더 선호하는지 결정하는 역할을 합니다. 연구진은 이 중심점들을 활용해 **삼각함수 급수(Trigonometric series)**로 토큰 간의 거리 선호도를 수학적으로 계산해 냈습니다.

#### ③ 성능 및 시사점

단순히 무식하게(brute-force) 토큰을 버리는 것이 아니라, 거리 선호도와 Q/K의 크기(Norm)를 기반으로 원칙적으로(principled) 중요도를 평가하여 압축합니다.

- **결과:** AIME25 같은 고난도 추론 벤치마크에서 32K 길이의 문서를 처리할 때, **정확도를 100% 유지(Full Attention과 동일)하면서도 KV 메모리 사용량을 무려 10.7배나 줄이거나, 처리량을 2.5배에서 최대 6.3배까지 향상**시켰습니다.
    
- **AI 인프라적 의미:** 수백 페이지의 기업 감사보고서를 처리할 때 메모리 부족(OOM)으로 거대한 고가 GPU 서버(H200 등)가 여러 대 필요했던 것을, 일반 소비자용 GPU 한 대나 저사양 서버에서도 돌아가게 만들 수 있는 게임 체인저입니다.

### 3. FlashAttention vs TriAttention 벤치마크 실습 가이드

두 기술은 상호 배타적인 것이 아니라 **보완적**입니다.

- **FlashAttention**은 GPU 메모리(HBM)에서 코어(SRAM)로 데이터를 퍼 나르는 '속도와 과정'을 최적화합니다.
    
- **TriAttention**은 저장해야 할 KV 캐시 데이터의 '양(Size)' 자체를 수학적으로 쳐냅니다.
    

이 두 가지를 비교/테스트하기 위한 실습 환경 구성 및 스크립트 작성 가이드입니다.

#### 실습 목표

32K 이상의 긴 기업 재무 데이터(Dummy data)를 입력했을 때, ①최대 VRAM 사용량과 ②초당 토큰 생성 속도(Throughput)를 측정합니다.

#### Python / PyTorch 기반 벤치마크 의사 코드 (Pseudo-code)

Python

```
import torch
import time
# (가정) 2026년 기준 triattention 라이브러리와 flash_attn이 설치된 환경
from triattention import apply_triattention_kv_compression
from flash_attn import flash_attn_func

# 1. 테스트 환경 셋업 (여신심사용 초장문 데이터 모사)
batch_size = 1
seq_len = 32768  # 32K 긴 문맥
num_heads = 32
head_dim = 128

# 랜덤 Q, K, V 텐서 생성 (FP16)
q = torch.randn(batch_size, seq_len, num_heads, head_dim, dtype=torch.float16, device="cuda")
k = torch.randn(batch_size, seq_len, num_heads, head_dim, dtype=torch.float16, device="cuda")
v = torch.randn(batch_size, seq_len, num_heads, head_dim, dtype=torch.float16, device="cuda")

def measure_performance(func, name):
    torch.cuda.empty_cache()
    torch.cuda.reset_peak_memory_stats()
    
    # 웜업 (Warm-up)
    for _ in range(3):
        _ = func()
    
    # 실행 시간 측정
    torch.cuda.synchronize()
    start_time = time.time()
    
    for _ in range(10):
        output = func()
        
    torch.cuda.synchronize()
    end_time = time.time()
    
    avg_time = (end_time - start_time) / 10
    peak_mem = torch.cuda.max_memory_allocated() / (1024**3) # GB 단위 변환
    
    print(f"[{name}]")
    print(f" - 응답 시간: {avg_time:.4f} 초")
    print(f" - 최대 VRAM 사용량: {peak_mem:.2f} GB\n")

# 2. 벤치마크 실행

# 케이스 A: FlashAttention만 적용 (빠르지만 32K 전체 KV를 유지해야 함)
def run_flash_attention():
    return flash_attn_func(q, k, v)

# 케이스 B: TriAttention 적용 (KV 캐시를 1/10 로 압축 후 Attention 수행)
def run_triattention():
    # TriAttention 알고리즘을 통해 K, V를 압축 (예: 32K -> 3.2K로 감소)
    compressed_k, compressed_v = apply_triattention_kv_compression(q, k, v, compression_ratio=0.1)
    # 압축된 텐서로 연산
    return flash_attn_func(q, compressed_k, compressed_v)

print("=== 32K Context Benchmark (FlashAttention vs TriAttention) ===")
measure_performance(run_flash_attention, "FlashAttention (Full KV)")
measure_performance(run_triattention, "TriAttention (10% Compressed KV)")
```

**실무 적용 팁:** 실제 사내 인프라에 적용하실 때는 오픈소스 추론 프레임워크인 `vLLM`이나 `llama.cpp` 에 TriAttention 플래그(`--triattention-budget` 등)를 켜고, 기업 심사 보고서 원문을 프롬프트로 밀어 넣어 서버가 죽지 않고(OOM 방지) 답변을 내놓는지 검증하는 과정이 필요합니다.
