### 1. 연산 최적화 (Computation Optimization)

GPU의 VRAM 사용량을 줄이고 메모리 병목을 해소하여 추론(Inference) 속도를 극대화하는 기술입니다.

#### ① FlashAttention

- **개요 및 역할:** LLM의 핵심인 어텐션(Attention) 연산 시, GPU의 느린 고대역폭 메모리(HBM) 대신 빠른 정적 램(SRAM)에서 연산을 타일(Tile) 단위로 처리하여 메모리 읽기/쓰기 횟수를 최소화하는 하드웨어 친화적 알고리즘입니다.
    
- **AI Infra 효과:** 연산 속도가 비약적으로 상승하며, 메모리 사용량을 O(N²)에서 O(N)으로 줄여 훨씬 긴 컨텍스트(입력 길이)를 처리할 수 있게 합니다.
    
- **금융업 시사점:** 수십 페이지에 달하는 대출 약관, 기업 재무제표, 펀드 설명서 등을 한 번에 입력하고 요약 및 분석하는 '문서 기반 AI 어시스턴트' 구현을 가능하게 합니다.
    
- **리스크:** 지원하는 GPU 아키텍처(Ampere 이상)가 필요하며, 레거시 장비에서는 적용이 제한될 수 있습니다.

	### **기초 지식: Attention이란 무엇이며 왜 LLM의 핵심인가?**
	
	LLM(대규모 언어 모델)은 기본적으로 '다음에 올 단어'를 예측하는 모델입니다. 이때 문맥을 이해하기 위해 사용하는 핵심 메커니즘이 바로 **어텐션(Attention)**입니다.
	
	- **개념:** 문장 내의 특정 단어가 다른 어떤 단어들과 가장 깊은 연관성을 가지는지(어디에 '집중'해야 하는지)를 수학적으로 계산하는 과정입니다.
	    
	- **작동 원리 (Q, K, V):** 검색 시스템과 유사합니다. 내가 찾고자 하는 질문(Query)을 데이터베이스의 키(Key)들과 비교하여 유사도를 구하고, 그 유사도를 바탕으로 실제 값(Value)을 조합해 냅니다.
	    
	    $$Attention(Q, K, V) = softmax(\frac{QK^T}{\sqrt{d_k}})V$$
	    
	- **왜 핵심인가?** 이 연산 덕분에 LLM은 수천 단어 떨어진 곳에 있는 정보도 잊지 않고 문맥을 연결할 수 있습니다. 하지만 문제는 **문장 길이(N)가 길어질수록 연산량과 메모리 사용량이 기하급수적으로 증가($O(N^2)$)한다는 점**입니다.
	    
	
	### **기술의 발전: 표준 Attention에서 FlashAttention으로**
	
	기존 Attention 연산의 가장 큰 병목은 'GPU의 연산 속도'가 아니라 **'메모리 접근 속도(Memory Wall)'**에 있었습니다.
	
	- **기존 방식의 한계:** GPU에는 엄청나게 큰 용량의 HBM(고대역폭 메모리)과 용량은 작지만 엄청나게 빠른 SRAM이 있습니다. 기존 Attention은 계산 중간중간 발생한 거대한 행렬 데이터를 계속 HBM에 썼다가 다시 읽어와야 했습니다. 이 읽기/쓰기 대기 시간 때문에 정작 계산을 담당하는 코어는 놀고 있는 현상이 발생했습니다.
	    
	- **FlashAttention의 혁신 (Tiling & Recomputation):**
	    
	    1. **Tiling(타일링):** 거대한 행렬을 HBM에서 한 번에 가져오는 대신, SRAM의 크기에 맞게 작은 블록(Tile)으로 쪼개서 가져와 SRAM 내부에서 연산을 끝내버립니다.
	        
	    2. **결과:** HBM을 오가는 횟수를 획기적으로 줄여, 메모리 접근 병목을 해소했습니다. 그 결과 **연산 속도는 2~4배 빨라지고, 메모리 사용량은 $O(N)$으로 줄어들어** 10만 토큰 이상의 긴 문서를 처리하는 최신 모델들의 등장을 가능하게 했습니다.
	        
	**하드웨어 지원 및 2026년 예산/도입 가이드**

	FlashAttention은 하드웨어 수준(SRAM)의 최적화가 들어가므로, 이를 지원하는 아키텍처의 GPU가 필요합니다. 2026년 우리은행의 사내 구축(On-Premise) 환경을 가정할 때의 가이드입니다.

|**GPU 아키텍처**|**대표 모델**|**FlashAttention 지원**|**특징 및 추천 용도**|**예상 도입 비용 (서버 1대, 8-GPU 기준)**|
|---|---|---|---|---|
|**Ampere**|A100|지원 (제한적)|과거의 표준. 현재는 감가상각이 많이 됨.<br><br>  <br><br>간단한 내부 PoC 및 개발/테스트용.|약 1.5억 ~ 2억 원|
|**Hopper**|H100, H200|**완벽 지원 (최적화)**|**현재 금융권 상용 서비스의 표준.**<br><br>  <br><br>H200은 메모리가 커서 긴 문맥 처리에 유리.|약 4.5억 ~ 6억 원|
|**Blackwell**|B100, B200|**완벽 지원 (최고 성능)**|2025~2026년 도입되는 최신 모델.<br><br>  <br><br>대규모 고객 대면 서비스 및 초대형 모델용.|약 6.5억 ~ 8억 원 이상|

> **💡 금융권 예산 시사점:** 은행의 인프라 구축 시에는 단순히 GPU 서버 비용뿐만 아니라, 이 엄청난 전력을 감당할 전력 설비, 쿨링 시스템(수냉식 등), 랙(Rack) 개조 비용, 고속 네트워크(InfiniBand) 스위치 비용이 서버 비용과 맞먹게 들어갑니다. 따라서 TCO(총 소유 비용)를 산정할 때는 GPU 하드웨어 비용의 1.5~2배를 예산으로 잡아야 합니다.

### 4. 실무 적용 가이드 (인프라 설정 및 코드)

AI 인프라 엔지니어가 직접 CUDA 커널 코드를 짤 일은 거의 없습니다. 대신, FlashAttention이 내장된 프레임워크를 최적으로 구동하는 환경을 구축합니다.

**① PyTorch 수준에서의 적용 (모델 개발 파트 지원 시)**

PyTorch 2.0 이상부터는 `F.scaled_dot_product_attention` 함수를 호출하면 하드웨어 환경을 인식하여 자동으로 FlashAttention을 적용합니다.

Python

```
import torch
import torch.nn.functional as F

# GPU에 텐서 할당
query = torch.randn(batch_size, num_heads, seq_len, head_dim, device="cuda", dtype=torch.float16)
key = torch.randn(batch_size, num_heads, seq_len, head_dim, device="cuda", dtype=torch.float16)
value = torch.randn(batch_size, num_heads, seq_len, head_dim, device="cuda", dtype=torch.float16)

# 별도 설치 없이 PyTorch 2.0+ 에서는 자동으로 최적의 Attention (FlashAttention 등) 선택
output = F.scaled_dot_product_attention(query, key, value)
```

**② 추론 서버 (vLLM) 인프라 구축 시 적용**

실제 은행 서비스에서는 vLLM, Triton Inference Server 등을 사용합니다. vLLM은 지원되는 GPU(Ampere 이상)에서 FlashAttention(현재 버전 2 또는 3)을 **기본적으로 활성화**합니다. 인프라 엔지니어는 환경 격리와 올바른 버전 관리에 집중해야 합니다.

Bash

```
# 1. 의존성 충돌을 막기 위해 Docker 컨테이너 환경에서 실행 (권장)
docker run --gpus all -v /data/models:/data -p 8000:8000 --ipc=host \
  vllm/vllm-openai:latest \
  --model /data/Llama-3-70b-Instruct \
  --tensor-parallel-size 4 \
  --dtype bfloat16

# * 참고: vLLM 구동 로그를 보면 "Using FlashAttention-2" 와 같은 메시지가 출력되며 활성화 여부를 확인할 수 있습니다.
```

### 5. 벤치마크 및 성능 테스트 방법

인프라를 구축했다면, 실제로 이 기술이 은행의 요구 성능(SLA)을 만족하는지 증명해야 합니다.

- **주요 측정 지표:**
    
    - **TTFT (Time To First Token):** 고객이 질문 후 첫 글자가 화면에 찍히기까지의 시간. (챗봇의 체감 속도 결정)
        
    - **TPOT (Time Per Output Token):** 글자가 1개씩 생성되는 속도.
        
    - **Throughput:** 1초에 시스템이 처리하는 총 토큰 수. (인프라 효율성)
        
- **성능 테스트 실행 (vLLM 내장 벤치마크 툴 사용 예시):**
    

Bash

```
# 가상의 금융 상담 데이터셋을 이용해 처리량 벤치마크 수행
python -m vllm.benchmarks.benchmark_throughput \
    --model /data/Llama-3-70b-Instruct \
    --tensor-parallel-size 4 \
    --input-len 2048 \
    --output-len 512 \
    --num-prompts 1000
```

이 벤치마크를 FlashAttention을 강제로 껐을 때(기본 Attention)와 켰을 때 비교해보면, 메모리 피크(Peak)가 낮아지고 Throughput이 급상승하는 것을 수치로 확인할 수 있습니다. 이것이 경영진에게 제출할 훌륭한 도입 근거 리포트가 됩니다.



#### ② Quantized KV Cache

- **개요 및 역할:** LLM이 단어를 생성할 때마다 이전 문맥을 기억하기 위해 저장하는 'KV 캐시'의 데이터 크기를 압축(예: FP16 -> INT8/INT4)하는 기술입니다.
    
- **AI Infra 효과:** 추론 시 VRAM의 가장 큰 비중을 차지하는 KV 캐시 크기를 대폭 줄여, 동일한 GPU에서 더 큰 배치 사이즈(Batch Size)를 처리할 수 있게 합니다. (Throughput 극대화)
    
- **금융업 시사점:** 뱅킹 앱을 사용하는 수백만 고객의 동시 접속을 감당해야 하는 B2C 챗봇 서비스에서, 서버 증설 없이도 더 많은 고객의 세션을 유지하고 응답할 수 있어 비용 절감에 직결됩니다.
    
- **리스크:** 과도한 양자화는 모델의 답변 품질(정확도) 저하를 일으킬 수 있습니다. 숫자에 민감한 금융 서비스에서는 할루시네이션(환각)을 유발할 수 있으므로, 정확도 손실률에 대한 엄격한 테스트가 필요합니다.
    

#### ③ Chunked Prefill

- **개요 및 역할:** 긴 프롬프트를 한 번에 처리(Prefill)하지 않고, 여러 개의 작은 청크(Chunk)로 나누어 다른 요청의 디코딩(단어 생성) 단계와 교차로 실행하는 기법입니다.
    
- **AI Infra 효과:** 긴 입력이 들어왔을 때 GPU 자원을 독점하여 다른 사용자의 응답이 지연되는 현상을 방지하고, 첫 토큰 생성 시간(TTFT)의 편차를 줄입니다.
    
- **금융업 시사점:** 대규모 문서를 분석하는 내부 직원의 요청과, 잔액 조회를 묻는 고객의 짧은 요청이 동시에 들어왔을 때 서비스 공정성을 유지합니다. 고객 서비스의 응답 지연(Latency) 불만을 방지합니다.
    
- **리스크:** 청크 사이즈를 최적화하는 스케줄링 로직이 복잡하며, 프롬프트 처리의 절대적인 시간 자체는 약간 늘어날 수 있습니다.
    

#### ④ llm-compressor (모델 경량화 도구)

- **개요 및 역할:** 가중치 양자화(Quantization), 가지치기(Pruning) 등을 통해 LLM 모델 자체의 크기를 줄이는 최적화 라이브러리/도구입니다.
    
- **AI Infra 효과:** 70B 이상의 거대 모델을 1/2 ~ 1/4 크기로 압축하여, 여러 대의 고가 GPU가 필요한 모델을 1~2대의 GPU 또는 저사양 GPU에 적재할 수 있게 합니다.
    
- **금융업 시사점:** 금융권은 데이터 외부 유출 방지(망분리)를 위해 온프레미스(사내 구축형) 환경을 선호합니다. 모델 크기를 줄이면 막대한 서버 구축(CAPEX) 비용을 절감하면서도 자체 보안망 내에서 AI를 운영할 수 있습니다.
    
- **리스크:** 압축 과정에서 금융 도메인의 전문 지식이나 복잡한 추론 능력이 손상될 수 있으므로, 파인튜닝(Fine-tuning)과 연계한 품질 검증이 필수적입니다.

### 2. 스케줄링 최적화 (Scheduling Optimization)

요청된 작업들을 어떻게 묶고 순서를 정해 GPU에 할당할 것인가를 다루는 기술입니다.

#### ⑤ Multi-step Scheduling

- **개요 및 역할:** CPU가 GPU에게 매 토큰(단어)을 생성할 때마다 명령을 내리는 대신, 한 번에 여러 스텝의 연산을 미리 스케줄링하여 던져주는 방식입니다. (CUDA Graph 등과 연계)
    
- **AI Infra 효과:** CPU-GPU 간의 통신 오버헤드와 PCIe 대역폭 병목을 줄여 GPU의 순수 연산 가동률을 극대화합니다.
    
- **금융업 시사점:** 장 마감 후 야간에 수만 건의 여신 심사 데이터를 일괄 전처리하거나, 일일 리스크 분석 리포트를 대량으로 생성하는 배치(Batch) 작업에서 처리 속도를 크게 단축시킵니다.
    
- **리스크:** 동적인 조건(사용자의 갑작스러운 요청 취소 등)에 유연하게 대응하기 어려울 수 있습니다.
    

#### ⑥ Continuous Batching

- **개요 및 역할:** 기존의 정적 배치(Static Batching)처럼 앞선 작업들이 모두 끝날 때까지 기다리지 않고, 작업이 끝나는 즉시 빈자리에 새로운 요청을 동적으로 밀어 넣는(In-flight) 스케줄링 기법입니다.
    
- **AI Infra 효과:** GPU 유휴 시간을 없애 전체 시스템의 처리량(Throughput)을 기존 대비 수십 배 향상시킵니다.
    
- **금융업 시사점:** 급여일, 공모주 청약일 등 트래픽이 폭증하는 이벤트 상황에서 AI 서비스가 다운되지 않고 안정적으로 요청을 처리할 수 있는 핵심 기술입니다.
    
- **리스크:** 빈번한 요청 교체로 인해 메모리 단편화가 발생하기 쉬워, PagedAttention과 같은 효율적인 메모리 관리 페이징 기법이 반드시 동반되어야 합니다.

### 3. 플랫폼 최적화 (Platform Optimization)

쿠버네티스(Kubernetes) 기반의 대규모 클러스터 환경에서 자원을 어떻게 분배하고 관리할 것인가에 대한 기술입니다.

#### ⑦ MIG (Multi-Instance GPU) / DAS

- **개요 및 역할:** MIG는 H100, A100 등 고성능 GPU 1개를 하드웨어 수준에서 여러 개의 작은 GPU(최대 7개)로 물리적 분할하는 기술입니다. (DAS는 빠른 로컬 데이터 접근을 위한 Direct Attached Storage 환경으로 해석할 수 있습니다.)
    
- **AI Infra 효과:** 크기가 작은 모델 여러 개를 구동할 때 GPU 낭비를 막고, 각 인스턴스 간 메모리/대역폭을 완벽히 격리하여 장애 전파를 막습니다.
    
- **금융업 시사점:** **보안과 격리**가 생명인 금융권에 적합합니다. 하나의 물리 서버에서 사내 부서용 AI와 고객 대면용 AI를 완벽히 격리된 자원으로 동시에 서비스할 수 있습니다.
    
- **리스크:** 분할된 자원은 고정되므로, 특정 인스턴스에 트래픽이 몰려도 남는 인스턴스의 자원을 유연하게 끌어다 쓰기 어렵습니다.
    

#### ⑧ Kueue (Kubernetes-native Job Queueing)

- **개요 및 역할:** 쿠버네티스 환경에서 다양한 AI/ML 작업(Job)들을 대기열(Queue)에 넣고, 우선순위와 할당량(Quota)에 따라 자원을 분배하는 잡 스케줄러입니다.
    
- **AI Infra 효과:** 여러 팀이 하나의 GPU 클러스터를 공유할 때 발생하는 자원 경합을 방지하고, 빈 공간에 작업을 효율적으로 욱여넣어 클러스터 전체의 가동률을 높입니다.
    
- **금융업 시사점:** 신용평가모델 팀, FDS(이상거래탐지) 팀, 마케팅 팀 등 여러 부서의 AI 학습/추론 요청을 우선순위(예: FDS 모델 업데이트는 1순위)에 따라 공정하고 안정적으로 배분할 수 있습니다.
    
- **리스크:** 큐잉 정책과 Quota 설정이 복잡하여 운영자의 쿠버네티스 숙련도가 매우 높아야 하며, 설정 오류 시 중요 작업이 무한 대기(Starvation) 상태에 빠질 수 있습니다.
    

#### ⑨ llm-d gateway (LLM API Gateway)

- **개요 및 역할:** 애플리케이션과 여러 LLM(오픈소스, 상용 API 등) 사이에 위치하여 로드밸런싱, 토큰 사용량 제어(Rate Limiting), 라우팅을 수행하는 중앙 관문입니다.
    
- **AI Infra 효과:** 인프라의 복잡성을 숨기고 엔드포인트 단일화, 캐싱을 통한 GPU 호출 최소화, 모델 장애 시 예비 모델로 우회(Fallback)하는 고가용성을 확보합니다.
    
- **금융업 시사점:** **가장 중요한 컴플라이언스 도구**입니다. 고객의 프롬프트에서 주민번호, 계좌번호 등 개인정보(PII)를 모델에 들어가기 전에 마스킹하고, 모든 AI 사용 이력을 감사(Audit) 목적으로 로깅할 수 있습니다.
    
- **리스크:** 게이트웨이 자체가 단일 장애점(SPOF)이 될 수 있으며, 트래픽 병목 지점이 될 수 있으므로 이중화 및 스케일아웃 설계가 필수입니다.
    

#### ⑩ DRA (Dynamic Resource Allocation)

- **개요 및 역할:** 쿠버네티스에서 기존의 제한적인 Device Plugin을 대체하여, GPU와 같은 특수 자원을 더 세밀하고 동적으로 프로비저닝/할당하는 새로운 프레임워크입니다.
    
- **AI Infra 효과:** 파드(Pod) 단위의 정적인 GPU 할당을 넘어, 필요할 때 네트워크 통신으로 다른 노드의 자원을 묶어 쓰거나 자원 라이프사이클을 세밀하게 제어할 수 있습니다.
    
- **금융업 시사점:** 망분리된 사내 프라이빗 클라우드 환경에서도 마치 퍼블릭 클라우드(AWS, GCP)처럼 유연하고 탄력적인 AI 자원 인프라를 구축할 수 있게 해줍니다.
    
- **리스크:** 도입 단계의 기술로 벤더(NVIDIA 등)의 드라이버 지원 및 생태계 성숙도를 지속적으로 모니터링해야 합니다.

### AI Infra 엔지니어로서의 준비 조언

위의 기술들은 단순히 "설치하고 쓴다"를 넘어, 각 요소들이 유기적으로 결합될 때 폭발적인 성능 향상을 이끌어냅니다. (예: _llm-compressor로 모델을 줄이고 -> vLLM에서 Continuous Batching과 Quantized KV Cache를 켜고 -> llm-d gateway로 트래픽을 제어_)

금융권 AI 인프라 엔지니어는 단순한 최고 성능(SOTA) 달성보다,
**①정확성(무결성) 
②보안/규제 준수 
③비용 효율(ROI)** 사이의 균형을 잡는 아키텍처를 설계할 수 있어야 합니다. 

위 개념들을 바탕으로 실제 오픈소스(vLLM, Ray, Kubernetes) 기반의 테스트베드를 구축해 보시는 것을 적극 권장합니다.
